## 两个看似不相关的 bug

pnpm-pub 用 OpenTray 做了一个托盘工具。用户点右下角图标，一个没有标题栏的面板弹出来，后面是 Windows 11 的 Acrylic 材质，前面是 Web UI。结构听上去很顺：原生窗口负责材质，WebView2 负责按钮和内容，各画各的。

但两个 bug 几乎同时出现。

第一个：resize 窗口时，刚才的按钮、文字甚至整块白色背景会沿着拖动轨迹留在窗口里，像一叠没清掉的截图，越拖越多。

第二个：窗口最小化以后，托盘菜单还写着 `Hide Window`。用户看到会困惑——窗口明明不在眼前，为什么还能 Hide？

代码层面这两个问题没有任何交集。一个是绘制问题，一个是文案问题。但追下去，它们都指向同一件事：在 Windows 桌面上，"透明"和"可见"到底是什么意思。

## 残影篇：把 WebView 缩到九分之一

最初很容易怀疑 WebView2。毕竟残留下来的正是网页内容。我们改过 WebView2 背景，试过 framed 和 frameless，也触发过 `WM_PAINT`、`RedrawWindow`、`UpdateWindow`。消息计数一直增加，画面却不肯变。

把方向扭过来的，是一个朴素的实验。我们把 WebView 的宽和高都改成窗口的三分之一，让它只占总面积的九分之一。剩下八分之九直接显示原生 Acrylic。

如果残影属于 WebView2，它至少应该被限制在那一小块区域里。结果不是。WebView 外面的 Acrylic 区域照样留下旧内容。

![背景残影：窗口调整尺寸后，旧页面内容沿着窗口移动轨迹重复出现](/assets/article-0062/background-residue.png)

这个结果排除了 DOM、CSS 和 WebView 背景。问题在更下面，至少已经到了原生窗口这一层。

### HWND 和 DWM

Windows 里的原生窗口由一个 HWND 标识，分客户区和非客户区。客户区是应用放内容的地方；非客户区是标题栏、边框和 resize 区域。

应用通过 WndProc 接收窗口消息（`WM_SIZE`、`WM_PAINT`、`WM_ERASEBKGND`），画完以后像素通常不直接进显示器。Windows Vista 以后，桌面由 DWM（Desktop Window Manager）统一合成。DWM 类比 Android 的 SurfaceFlinger、macOS 的 WindowServer，或者浏览器里负责合成图层的 compositor。为了做到这些，DWM 会为每个窗口保留离屏内容，工程里叫 redirection surface（重定向表面）。

OpenTray 的结构又多了一层。WebView2 不是在父窗口里执行绘图函数，它会创建自己的 child HWND。于是实际层级是：

```text
Windows 桌面
└─ DWM 负责最终合成
   └─ OpenTray 顶层 HWND
      ├─ 非客户区：标题栏、边框、resize frame
      └─ 客户区重定向表面
         ├─ OpenTray 原生宿主底层
         └─ WebView2 child HWND
            └─ HTML 页面
```

前端里的 `background: transparent` 只意味着 HTML 这一层不提供背景像素。它不会替父 HWND 生成内容，也不会替 DWM 修复一张旧表面。

### Acrylic 不是 CSS 背景

Acrylic 和 Mica 是 DWM 提供的 system backdrop。应用告诉 DWM 想用哪种材质，DWM 根据系统主题和桌面内容生成效果。它们不是网页的 `backdrop-filter`，也不是 WebView2 画出来的半透明图片。

这意味着材质窗口仍然需要一个完整的原生客户区。WebView2 透明以后，DWM 材质从下面透出来；如果父 HWND 从来没有稳定拥有那块表面，DWM 就可能继续拿旧内容参与合成。

为了验证，我们做了 `native-material-host-paint-probe-20260716.exe`。探针只保留原生窗口、Acrylic/Mica 切换和三种 host paint：

| 模式 | 原生窗口做了什么 | 结果 |
|---|---|---|
| No host paint | 收到绘制消息，但不填客户区 | resize 后留下旧内容 |
| Black host paint | 用黑色画刷填满客户区 | Acrylic 仍然可见，残影消失 |
| Gray host paint | 用灰色画刷填满客户区 | 确认父窗口绘制确实参与合成 |

第二种模式很反直觉：父窗口刷黑，为什么窗口没变黑？因为黑色是 DWM 材质的合成底，不是盖在 WebView2 上的黑色图层。Acrylic 仍由 DWM 呈现，WebView2 仍只画按钮和文字。

这也解释了为什么 `Invalidate + UpdateWindow` 没用。`WM_PAINT` 计数增加，只能证明 WndProc 被调用；处理函数没为目标区域提供有效内容，消息再多也不会得到正确像素。

OpenTray 的材质窗口现在在 `WM_ERASEBKGND` 和 `WM_PAINT` 中用 `BLACK_BRUSH` 填满客户区：

```text
WndProc(hwnd, message):
  if isMaterialWindow(hwnd) and message is WM_ERASEBKGND:
    FillRect(hwnd.clientRect, BLACK_BRUSH)
    return handled
  if isMaterialWindow(hwnd) and message is WM_PAINT:
    paint = BeginPaint(hwnd)
    FillRect(hwnd.clientRect, BLACK_BRUSH)
    EndPaint(hwnd, paint)
    return handled
  return DefWindowProc(hwnd, message)
```

### 什么时候画，和画什么一样重要

两个 Win32 style 容易踩坑。`WS_CLIPCHILDREN` 让父窗口绘制时避开子窗口区域——子窗口不透明时这是优化，WebView2 透明时它等于让父 HWND 放弃 WebView2 下面的区域，所以材质窗口需要移除它。`WS_EX_NOREDIRECTIONBITMAP` 表示窗口不要求 DWM 保留通常的重定向表面，材质窗口却需要 DWM 宿主表面，把普通透明窗口的 style 原样搬过来不会稳定。

冷启动顺序也关键。WebView2 controller 创建时会生成 child HWND、初始化 COM、触发一批尺寸消息。如果父窗口还没完成材质、初始尺寸和 host paint，第一张进入 DWM 的表面就可能不完整。

OpenTray 的冷启动顺序：

```text
createMaterialWebView(options):
  hwnd = createHiddenTopLevelWindow()
  setHostPaintPolicy(hwnd, black)
  applyWin32Style(hwnd, options.style)
  applyDwmBackdrop(hwnd, options.material)
  setInitialBounds(hwnd, options.bounds)
  recommitNativeHostSurface(hwnd)
  context = createWebViewContext(stableProfilePath(options))
  webview = createWindowedWebView2(hwnd, context, transparent=true)
  initializeAppWindowOverlay(hwnd)  // WebView2 已建立 COM
  setWebViewBounds(webview, getClientRect(hwnd))
  showWindow(hwnd)
```

resize 也不能反过来。如果先放大 WebView2 再补画父 HWND，新露出的区域会先显示父窗口上一轮的重定向内容。OpenTray 先交给 `DefWindowProc` 处理系统几何，再提交父表面，最后更新 WebView2：

```text
on WM_SIZE:
  DefWindowProc(hwnd, WM_SIZE, ...)
  recommitNativeHostSurface(hwnd)
  bounds = getClientRect(hwnd)
  webview.controller.setBounds(bounds)
  moveChildWindow(webview.hwnd, bounds)
  webview.controller.notifyParentWindowPositionChanged()
```

### 非客户区的边框残影

客户区残影消失后，又看到一圈老式窗口边框。标准 Windows 窗口带 `WS_OVERLAPPEDWINDOW`、`WS_THICKFRAME` 等 style，Windows 据此生成标题栏和边框。frameless 可以彻底移除原生 frame，也可以保留 `WS_THICKFRAME` 的缩放能力，再通过 `WM_NCCALCSIZE` 扩大客户区。

![非客户区边框残影：frameless 窗口外侧残留旧 Windows 边框](/assets/article-0062/non-client-frame-residue.png)

一次回归里，左右只多 1px，顶部却多了 7px：原生窗口高度 506，页面的 `window.outerHeight` 只有 499。那 7px 不是 CSS margin，而是 caption 还占着位置。修复时保留左右和底部的原生 resize inset，顶部只保留 `DWMWA_VISIBLE_FRAME_BORDER_THICKNESS`。

### WebView2 还是一个浏览器

窗口显示正确后，pnpm-pub 又暴露了发布路径问题。本地 `pnpm pub start` 正常，`pnpx pnpm-pub@latest start` 却只显示托盘图标几秒。原因是 WebView2 的 user data folder（浏览器 profile）跟随 exe 位置，pnpx 把可执行文件放在临时目录，路径变深、权限变化或锁冲突时 WebView2 环境创建失败。现在 OpenTray 使用稳定目录 `~/.opentray/webview/<version>/<caller>`。

### 残影排查路径

```text
屏幕上的异常像素
├─ DOM 与 CSS
├─ WebView2 surface 或 child HWND
├─ 顶层 HWND 客户区与 DWM redirection surface
├─ 标题栏、边框、resize frame 等非客户区
└─ COM、profile、broker 与安装路径
```

消息计数不能代替视觉结果。原生对照程序也必须和真实程序共享窗口大小、DPI、材质、style 和创建顺序，每次只改一个变量。

这次之后，我对 `transparent` 这个词谨慎了很多。在浏览器里它通常只是露出父元素；在 Windows 桌面里，它会一路露出 WebView2 的父 HWND、DWM 材质和系统保留的窗口表面。父窗口没画好时，透明层不隐藏问题，它把那张旧表面老老实实露出来。

## 生命周期篇：隐藏了还算开着吗

回到第二个 bug。窗口最小化了，托盘菜单还写着 `Hide Window`。加一个 `isOpen = false` 不就行了？

麻烦在 `isOpen` 这个词。一个托盘窗口被最小化了，它算开着吗？被隐藏但 WebView 还在，它算开着吗？窗口失焦后自动收起，和用户主动关闭，是不是同一种状态？

如果没先说清楚，Show/Hide 文案只是第一个出错的地方。后面还会遇到重复创建窗口、页面状态丢失、任务栏多出图标，以及同一个 broker 被反复订阅。

### 三个对象，不是一个

普通桌面应用通常把主窗口当成应用本身。窗口关了，应用结束。托盘应用不是这样。OpenTray 里至少有三个对象：

```text
托盘应用
├─ Tray icon：长期存在，接收点击
├─ Native window：显示、隐藏、最小化、置顶
└─ WebView session：页面状态、Cookie、脚本运行时
```

用户隐藏窗口时，通常只想收起面板，不想重新登录，也不想丢掉刚才的工作区。因此 OpenTray 使用 retained session：窗口暂时不显示，但 HWND、WebView 和页面状态继续存在。

### show() 只做第一次

第一次打开窗口时要创建原生窗口、装载 WebView、注册事件再显示。这套动作由 `show()` 完成，更接近 bootstrap，不是可反复调用的显示开关。会话建立后动作分开：

```text
show()       第一次创建并装载窗口会话
toVisible()  恢复已隐藏或最小化的窗口
close()      隐藏窗口，但保留 WebView 会话
destroy()    释放原生窗口和 WebView
```

每次点击托盘都重新 `show()`，页面会重复装载，listener 会重复注册。每次 `close()` 都销毁窗口，WebView 页面和缓存又不断重建。所以托盘点击是"显示或隐藏同一个 retained session"，真正退出时才 `destroy()`。

### operational visibility

Windows 有 `IsWindowVisible` API，但它回答 style 层面的可见性，不完全等于用户能不能操作。一个最小化窗口可能仍属 visible window，却不在用户眼前。

OpenTray 使用 operational visibility：窗口没被隐藏、没最小化、用户现在可以操作它。

```text
operationalVisible = windowExists and not hidden and not minimized
```

托盘菜单关心这个状态。窗口最小化后菜单显示 `Show Window`；用户点击后调用 `toVisible()`。

### 原生窗口是事实源，页面不猜 HWND

WebView 页面有 `document.visibilityState` 和 blur 事件，但它们回答的是页面问题，不知道 HWND 是否最小化、被原生代码隐藏或还在任务切换器里。如果页面自己维护 `shown` 布尔值，页面以为窗口还显示着，Windows 已经最小化了。两份状态没谁能自动纠正谁。

OpenTray 让原生窗口成为事实源。应用查询 `isVisible()` 或订阅 `visibleChange`，最小化、原生 close、失焦 auto-hide 都收敛成同一种事件。托盘菜单和页面只是这个事实的投影。

```text
onNativeVisibilityChanged(visible):
  tray.primaryLabel = visible ? "Hide Window" : "Show Window"
  webUi.windowVisible = visible
```

### autoHide 是谁负责关门

多数托盘面板有个共同习惯：用户点到别处，窗口自动收起。OpenTray 把它叫 `autoHide`，默认由原生宿主处理。但一个明确要求置顶（`keepOnTop`）的窗口不该每次失焦自动消失——这两个设置在这个条件里产生交集。

pnpm-pub 是例外。它关闭窗口前要播页面动画，所以设 `autoHide: false`，页面完成动画后报告 `completeAutoClose`，daemon 再调原生 `close()`。页面负责动画，原生宿主负责窗口事实。

### broker connection closed 为什么重复打印

窗口状态最终要从原生 broker 传回 JavaScript。一次实现问题让每个 listener 都启动自己的轮询，几个请求同时读同一条连接。broker 关闭时这些重叠请求一起失败，终端反复打印 `broker connection closed`。看上去像窗口连续崩了，实际是同一个 transport 失败被多个轮询重复报告。

修复用 shared single-flight loop：

```text
drainLoop():
  if requestInFlight: return
  requestInFlight = true
  try:
    events = await broker.drainWindowEvents()
    dispatch(events)
  catch transportError:
    stopLoop()
    reportOnce(transportError)
  finally:
    requestInFlight = false
```

### 状态机

把上面的规则合在一起：

```text
未创建
  │ show()
  ▼
可操作可见
  ├─ close() ─────────────> 隐藏 retained session
  ├─ minimize() ──────────> 最小化 retained session
  ├─ blur + autoHide ─────> 隐藏 retained session
  └─ destroy() ───────────> 已销毁

隐藏或最小化
  ├─ toVisible() ─────────> 可操作可见
  └─ destroy() ───────────> 已销毁
```

每次状态变化由原生窗口计算 operational visibility，发出 `visibleChange`。托盘菜单更新下一次动作，页面更新显示投影，没有任何一层重新发明窗口状态。

最开始那个 `Hide Window` bug 不用单独打补丁了。菜单每次向原生窗口询问 operational visibility：看得见就显示 Hide，看不见就显示 Show。

以后再做 Electron、Tauri 或原生托盘应用，我会先画出托盘图标、原生窗口和页面会话，再问三件事：谁决定窗口是否可操作，谁负责保留页面状态，谁控制任务栏投影。框架名称会变，这三个边界不会跟着消失。
